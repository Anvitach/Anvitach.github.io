Chapter 1. Introduction

If you often receive any news or updates about vulnerabilities such as critical vulnerabilities being exploited by hackers, zero-day vulnerabilities or exploits, software vendors releasing patches and fixes, etc. and wonder what all this fuss is about then you are in the right place.
The core of this whole self-help guide and the one concept which the author is trying to make their readers understand and care about is vulnerabilities. In general English, a vulnerability means an exposure which can make the entity accommodating that exposure susceptible to harm. But what does it mean to have vulnerabilities in a computer system? Why is it important to learn about them? What are different sources of vulnerabilities and how should a beginner software developer handle them? These are few of the questions that author will try to answer in this self-help guide.

1.1. What is a vulnerability?

Let’s first start by addressing what does a vulnerability mean in a computer system.

Important
 A vulnerability is a weakness or a flaw in software or hardware which increases the likelihood of being exploited by an adversary to harm the said software or hardware.

In this guide, we will strictly restrict our scope to software vulnerabilities. Now let’s dig into the above definition. A flaw or weakness which increases the likelihood of being exploited: this means that there’s a possibility of that vulnerability not being exploited at all but it still increases the likelihood of being exploited. There are many vulnerabilities which don’t even get highly publicized like Heartbleed, Log4Shell, ProxyLogon, Psychic Signatures, etc. and get patched before the outside world even gets to know about them. If a vendor patches a vulnerability before it can be exploited then it is good, right? It is definitely better than a zero-day vulnerability (definition included below). Now, what is a zero-day vulnerability.

Important
 A zero-day vulnerability is a weakness or a flaw which is disclosed to the public before it grabs the attention of the software vendors.

They are called zero-day vulnerabilities because the software vendors have exactly zero-days to fix that vulnerability. As those vulnerabilities become public before the software vendors can mitigate them, it becomes harder and critical for those vendors to release patches before adversaries can exploit them.

But why such a sudden need to learn about vulnerabilities especially for beginner software developers? Since most of you are training to work in a DevOps environment, once you enter the corporate world you will be in one way or other contributing to code bases of softwares, applications, or products. And such code bases include vulnerabilities which we read in news. Of course, many DevOps teams have incorporated security approaches in their DevOps Lifecycle to get around these vulnerabilities such as DevSecOps, DevOpsSec, or SecDevOps but isn’t it better to incorporate such practices and make them habitual before we start making a contribution to such code bases.

When taken literally each term signifies where the security coding practices and security testing lie in a Software Development Life Cycle (SDLC). For DevSecOps, the secure coding practices are inculcated once the planning and development part is done but it preempts the deployment and operations part of SDLC. In DevOpsSec, the security review is conducted once the development, continuous integration and continuous deployment is finished. Further in SecDevOps, the secure practices are integrated into the workflow of development and deployment part. Most firms and companies adapt the mentioned practices above or practices that are analogous to them. Handling vulnerabilities becomes an essential part of this process regardless of the set of practices being chosen.

As we can see in Figure 1. <1> , we can notice a CVE ID which is given to each vulnerability to uniquely identify them. These CVE IDs are assigned to vulnerabilities by CVE Assignment team and/or by CVE Numbering Authorities (CNAs) who address vulnerabilities lying in their scopes. Figure 1. <2> addresses the description of each vulnerability which inculcates the vulnerability type, the product/software vender, and the affected code base(s). Moving on to Figure 1. <3> , it can be easily discerned that there are two versions of Common Vulnerability Scoring System (CVSS) used to score a vulnerability i.e. CVSSv2 and CVSSv3. Most of the vulnerabilities published before July 13, 2022 are scored using CVSSv2 after which it was discontinued by NVD. CVSSv3 was introduced to calculate base score after considering impact on confidentiality, integrity, and availability of the affected system. However, CVSSv3 was introduced in 2015 and many vulnerabilities since then are scored on basis of both CVSSv2 and CVSSv3 which is why Log4Shell has been allocated both CVSSv2 as well as CVSSv3 score. Figure 2. CVSSv2 and CVSSv3 Base Score Ranges

Next, Figure 1. <5> informs us about the Common Weakness Enumeration (CWE) categories in which a particular vulnerability lies. CWE is a list of types of weaknesses found in hardware and softwares. These weaknesses can be considered as the origin of vulnerabilities.

A CWE has nothing to do with a product or software or system. It can be common amongst multiple vulnerabilities and a vulnerability can belong to multiple CWEs. Finally, Figure 1. <6> represents Common Platform Enumeration (CPE) associated with the vulnerability. CPEs are naming format structure to identify specific platforms, applications, systems, etc. With the help of this, vendors of those products can identify which versions of their product are affected by that vulnerability. The naming format according to Acunetix’s Common Platform Enumeration website[7] is

cpe:2.3: <part>:<vendor>:<product>:<version>:<update>:<edition>:<language>

where
* part: Defines the type of system detected, and can have one of the following values:
a - for Application
h - for Hardware
o - for Operating System
* vendor: Contains the name of the organization that developed the product.
* product: Contains the name of the product that has been detected.
* version: Lists the version number for the product.
* update: Lists the update for the product and version detected.
* edition: Shows the edition of the software.
language: Shows the language detected.

In Figure 1. <6> one of the CPEs can be represented as [part = "a", vendor = "apache", product = "log4j", version = "2.0", update = "beta9", edition = "any variation", language = "English"]. Here * in edition means any variation in edition of that system is affected with that vulnerability. Also, the language is by default English.

Now that we’ve gathered enough information about CVEs, we have to understand why is the author stressing on this subject. If the author asked you to make a guess right now about how many vulnerabilities in total have been discovered as of now, how much would you guess? 10,000? 50,000? 100,000?

Let me tell you the current number of vulnerabilities, till the author finished this guide which is around January 2023 is 192,255. And around 57% of these vulnerabilities have been discovered from 2017-2022 as can be seen in Figure 3. and it has been increasing every year abruptly in that range.

In the past year, 18 vulnerabilities inhabited Microsoft Exchange Server out of which 2 of those were considered zero-day vulnerabilities which were used late in this year by attackers to perform Active Directory reconnaissance and data exfiltration[8]. Although Apple products didn’t experience any high-profile attacks, they still have come into 385 vulnerabilities in 2022 alone[9]. Not to mention, the 2 OpenSSL 3.0 vulnerabilities namely CVE 2022-3786 and CVE 2022-3602 published in early November, 2022 which affected 16,000 publicly accessible servers at that time[10].

The author understands that having vulnerabilities in a system is common but it is important to bring your attention into the fact that irrespective of vendors, products or SDLC approaches companies follow, vulnerabilities can still find their way into their system. In addition to that, a system’s security is as good as its weakest link and vulnerabilities literally are weaknesses. Through this, the author hopes that the reader has got hold of the CVE explosion scenario that the author is trying to state.

In an ideal world, the author would wish for computer systems to be developed without any vulnerabilities but as we all know; perfect security is unachievable. However, the author does believe that by arming beginner software developers, who’ll join such vendors in developing their product with a few vulnerability scanning, searching, and replacing tools/utilities along with a good background of vulnerability management will help those developers navigate through the SDLC approaches mentioned before. Moreover, this’ll also strengthen their decision making in vulnerability treatment as well as encourage them to perform risk analysis when they encounter such vulnerabilities.

1.4. Source of vulnerabilities

Vulnerabilities can turn up in a code base because of 2 reasons

  • Upstream components

  • In-house generation

1.4.1. Upstream components

Before we understand what upstream components are we need to first learn about SBOM.

Important
 Software Bill of Materials (SBOM) is a comprehensive list of components, packages, libraries, etc. used in order to create a code base.

This term has been adapted from Bill of Materials used in manufacturing which is a list of parts, raw materials and components used in building a product. Understanding SBOM is necessary because even though we might not have felt the need to understand about various building blocks of code bases yet, once we enter the industry of software or product development we will need to learn it one way or the other. During our degree programs, we are taught to create our own programs and type every line of our program code from the scratch in order to strengthen our programming skills; but when it comes to organizations, their products consist of thousand and million lines of code which might use multiple open source and third-party components. These third-party components could be reusable software units which are free and publicly licensed, or sold by third-party vendors, or developed by the same organization used in their past projects. For example, earlier author gave an example of OpenSSL 3.0 vulnerabilities which affected 16,000 servers. Here, OpenSSL 3.0 is the software library used in multiple applications which is why just 2 vulnerabilities in this library were able to affect so many systems. Such upstream components make it easier for adversaries to carry out supply-chain attacks.

Open Source Security and Risk Analysis (OSSRA) 2022 report[11] documents that 97% of the 2400 code bases which were analyzed contained open source components and 78% of these code bases was generated by open source components. There are many times that developers make use of open source components without even knowing about them which is why formulating a SBOM for code bases as well as frequently using vulnerability scanners on code bases is essential. The author will eventually present OWASP Dependency-Check and Intel CVE Binary tools which can be used for this purpose.

1.4.2. In-house generation

Once, we create SBOM and learn how to handle upstream vulnerabilities we are not even half way done with our vulnerability management practice. But before we move forward we need to learn what do bugs mean in a computer system

Important
 A bug is an error or defect in a software or hardware which cause the same to behave unexpectedly.

Again as the scope of this guide is limited to the software environment, we’ll only focus on software bugs. Vulnerabilities are merely a subset of bugs. These bugs can be caused by poor programming practices and/or poor design logic. In fact, Number of Faults per Line of Code[12] says that as the number of lines of code increases, the probability of number of faults is expected to increase as well. Further, their study indicated that a 100 lines of code can have a fault rate of 0.6 to 1.7. Of course, they are lots of variables that come into play while estimating fault rates such as the type of programming language used, the developer’s programming skill, complexity of the code, etc. but most of the studies circle around the above mentioned fault rate. If rounded off then it means that there’s a good probability of finding 0 to 2 bugs in 100 lines of code.

Many of the bugs which are published till now are results of poor secure programming practices or lack of proper code testing. For example, let’s have a look again at Figure 1. <5> . One of the CWEs listed there is CWE-20 i.e. Improper Input validation. More information about this CWE can be gathered from CWE MITRE website[13] but if loosely translated, inheritance of this weakness means that the input being transferred between different components of the code isn’t validated which can eventually be used for code execution. It is harder to discern whether this weakness was inherited by Apache Log4j2 by some other upstream component or generated by developers in Apache4j2’s code base but it is clear that is a result of poor secure programming practice carried by someone in the hierarchy of ApacheLog4j2 which in the end led to Log4Shell vulnerability.

All of these occurrences can be reduced by performing semantic as well as syntactic analysis of written code which should be made part of vulnerability management practice. The author will eventually present CodeQL analysis engine and Comby tool which can be used for this purpose.

Chapter 2. OWASP Dependency-Check Tool

What is OWASP dependency-check tool?

If one surfs the internet it is very easy to find that OWASP dependency-check is a free, open-source Software Composition Analysis (SCA) tool. SCA tools are used basically to identify open source and third-party components in a software code base by creating a SBOM. The tool in short, maps all open-source components with a developer’s source code and detects all possible components and packages used in a software build. SCA is part of Application Security Testing (AST) which only focuses on dependencies and not on the part of the code written by developers.

Such software components and dependencies need to be tracked and scanned in a code base because organizations' code bases are depending on said upstream third-party software components which also bring the possibility of harboring software vulnerabilities along with them. We as software developers need to develop a practice of scanning our code base for vulnerabilities in order to maintain code hygiene. SCA tools like OWASP dependency-check assist developers like us to identify those vulnerabilities and platforms that have brought those vulnerabilities in our code.

This tool was specifically chosen by the author because firstly, it is open-source and free tool because of which software development engineers wouldn’t have to pay for commercial tools. Secondly, OWASP Dependency-Check tool was one of pioneers in using SBOM for vulnerability scanning. If the readers wish to choose any other tools then they are free to do so but OWASP Dependency-Check can be considered as the author’s choice.

Let’s dive into the tool and its functionality by going back to our Running Example.

Anna Chadwick had once read about a free software scanning tool named OWASP dependency-check tool which is used to scan code bases for vulnerabilities by referencing the NVD. Being curious about the tool she goes and reads about the tool in Dependency-Check GitHub documentation website [15]. Guessing that the tool can serve as a means to look for open-source vulnerabilities in her project, she followed the simple instructions provided in the Dependency-Check GitHub repository[16] to install and run dependency-check on her working folder. The command she used to run OWASP dependency-check on her command line was -

Chapter 3. Intel CVE Binary Tool

Unlike OWASP Dependency-Check, CVE Binary tool is a binary scanner, also used to isolate and identify vulnerable open-source components. When organizations make use of third-party components such as containers or databases, etc. these kind of components which come in object forms don’t often have source codes which is why binary scanning of such files is necessary. Binary Scanners are adept in handling such situations. Moreover, binary scanners also procreate SBOM of a software’s final build before scanning for vulnerabilities.

The author chose this tool because again it is a free open-source tool. Another reason why the author chose this tool because it is backed by Intel. Backed by such a pioneer organization the owners of its repository have made it a point to keep updating that tool. In the past year, the author herself has seen this tool advance. A tool which was a binary scanner at first now uses SBOM to identify upstream libraries.

Chapter 4. CodeQL

The tools mentioned before will definitely help a beginner software engineer in upping their game in development when it comes to handling vulnerabilities inherited via upstream components. But what to do when the weaknesses behind such vulnerabilities are induced in the code written by the developers themselves i.e. if vulnerabilities are generated in the house? A CVE when gets published is tied to a software configuration or a product. Although the CVE gets published in a public database, it is possible for a developer to imitate a variant of that vulnerability in a software written by them. In such cases, it is much more important to understand the cause and effect of a vulnerability, rather than just identifying them according to their underlying CPEs and CVEs, etc. If shortly put, this means that the developers need to understand the semantics of their code and analyze them.

CodeQL semantic analysis engine is a suitable solution for the above mentioned problem. This engine converts a piece of code into a queryable database which is in the form of a hierarchical representation such as an abstract syntax tree and a data flow graph. CodeQL works as a supercharged grep which allows its users to extract variants of vulnerabilities through CodeQL query language which is more like a complex SQL-like syntax.

The author chose this engine for semantic analysis because author believes that it is very powerful when it comes to taint tracking and data flow within a code base. Moreover, this is a GitHub issued engine which can be integrated along with GitHub workflow. Thirdly, if integrated successfully, GitHub itself scans the repository everyday using default CodeQL queries created by GitHub itself. Finally, it is a free and open-source engine which is suitable for beginning software engineering developers. Now let’s learn about the engine through Anna’s perspective.

After coming across 2 new tools which Anna believed would help her make her project free of vulnerabilities, especially from upstream components, she was ready to continue working on her own code for the project. Comby is a tool which can be specifically used to search and replace lexical syntax to enhance the grammar of a code base. Its matching pattern is much better than regular expressions because it disregards the unbalanced entities for example, if matching is done for a block of multiple lines one of the line includes unbalanced parentheses in a comment. Moreover, it also accommodates matching using Regular Expressions inside Comby holes to make searching and matching better for Comby. Another added advantage of this tool is mass replacement of syntaxes. Comby enables its users to generalize multiple similar syntaxes and replace them all together.
